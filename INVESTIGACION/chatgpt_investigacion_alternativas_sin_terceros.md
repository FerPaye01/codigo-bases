# Resumen ejecutivo  
Para el **Asistente de Bases Estándar** y el **Asistente de Contratos**, proponemos una arquitectura escalable y segura combinando servicios gestionados de nube con componentes de código abierto. Para el MVP, recomendamos centrarse inicialmente en el flujo de *Bases*: extraer datos del TDR (términos de referencia) con OCR y NLP, rellenar plantillas `.docx` predefinidas y generar el documento final, siempre con revisión humana obligatoria. Se sugiere usar un orquestador **serverless** (p.ej. AWS Step Functions o Azure Logic Apps) para encadenar la consulta a SIGED, OCR y generación de documento, aprovechando servicios administrados que minimicen la infraestructura propia y aceleren el desarrollo. Para un piloto, se podría emplear **AWS Textract** o **Azure AI Document Intelligence** para OCR (ambos con buen soporte de español) y un modelo LLM en la nube (p.ej. Azure OpenAI GPT-4o o Amazon Bedrock con Claude) para extracción de campos con RAG, enfocando el desarrollo de reglas de validación simples.  

En producción, la recomendación es evolucionar hacia una solución híbrida que permita datos sensibles on-premise. Por ejemplo, usar **Azure Bot Service/Azure Functions** o **AWS Lambda/Lambda Durable Functions** según la plataforma elegida, y bases de datos seguras (PostgreSQL administrado o Cosmos DB) con cifrado. Para NLP y LLM, puede considerarse hospedar localmente modelos abiertos (Llama, Mistral, Qwen) en Kubernetes para cumplir restricciones de datos sensibles. En todo caso, se debe priorizar una **arquitectura trazable**: cada dato extraído debe referenciar el documento fuente y el campo de plantilla. Se utilizarán logs estructurados, auditoría de acciones y versionado de documentos (S3/Azure Blob con control de versiones).  

Finalmente, la solución debe incluir revisiones humanas obligatorias para evitar **alucinaciones** de la IA, con mecanismos de control como validación de reglas de negocio y valores límite (montos, plazos, datos obligatorios). El flujo de implementación en fases será: (1) **MVP Bases** – flujos básicos OCR/NLP + plantillas, (2) **Robustecimiento OCR/NLP y motor de plantillas**, (3) **Asistente de Contratos** – integración de múltiples documentos y validaciones avanzadas, (4) **Despliegue productivo** con observabilidad y auditoría completa. Este enfoque modular permite iterar rápidamente, validando componentes críticos de forma controlada.

## Arquitectura recomendada para MVP  
El MVP se centrará en el flujo **Bases Estándar**. Proponemos una arquitectura serverless en la nube que combine: (1) **Interfaz de usuario web** sencilla (p. ej. una app React con BotUI) para consulta de expediente y visualización de datos, (2) un **servicio orquestador** basado en AWS Step Functions o Azure Durable Functions que coordine el flujo, (3) **LLamadas a SIGED** mediante API (o simulación de datos si SIGED no está disponible), (4) **OCR/NLP** en la nube (AWS Textract o Azure Cognitive OCR + Azure OpenAI) para extraer información del TDR y generar campos, (5) **Motor de plantillas** (p.ej. docxtpl en un Lambda/Azure Function) que rellena la plantilla `.docx` correspondiente, y (6) **Almacenamiento** en S3/Blob Storage con URLs firmadas para descarga. Todas las acciones críticas quedan registradas (CloudWatch/Azure Monitor). La generación del documento puede ser síncrona (docx directo al usuario), con opción a convertir a PDF offline (usando un contenedor LibreOffice o Aspose) si es necesario. Los usuarios validan y corrigen los datos extraídos antes de generar el archivo final, garantizando control humano. Esta solución MVP prueba la viabilidad de OCR/NLP y plantillas con mínima infraestructura.   

## Arquitectura recomendada para producción  
En producción recomendamos un diseño híbrido y desacoplado:  
- **Frontend**: se mantiene la app web/chat, pero en un entorno seguro (VPC privada o subnet protegida) con autenticación corporativa (IAM/Azure AD). Se añaden controles de DLP para que documentos sensibles no abandonen el entorno.  
- **Backend/Orquestador**: adoptar microservicios en Kubernetes (EKS/AKS). El flujo de bases y contratos podría dividirse en microservicios especializados (p.ej. `service-ocr`, `service-extraccion`, `service-plantillas`), orquestados por Temporal o Step Functions Federation. Esto permite escalado y logging detallado. Todos los servicios deben comunicarse a través de colas (SQS o Event Grid) o APIs con autenticación interna (IAM roles/mSIs).  
- **SIGED**: en entornos estrictos, podríamos desplegar un adaptador local (RPA en VM gestionada) que sincronice diariamente nuevos expedientes si no hay API en tiempo real. En la nube, integrar SIGED vía API REST con una capa de encriptación para cumplir políticas.  
- **Persistencia**: usar bases de datos gestiona- das: p.ej. PostgreSQL en RDS (o Azure PostgreSQL) para metadatos, DynamoDB/Aurora Serverless para estados de flujos. Para búsquedas RAG, un motor vectorial (Elasticsearch Service o Weaviate on-prem) con metadatos enlazados a la fuente.  
- **OCR/NLP**: en nube se puede continuar con servicios SaaS, pero con versión administrada de OCR (Textract async, Azure Form Recognizer Custom) y modelos LLM. Para documentos extremadamente sensibles, habilitar un nodo on-prem (GPU) con modelo abierto fine-tuneado, ejecutándose en EKS privado.  
- **Seguridad**: habilitar cifrado KMS/Key Vault en todos los datos, VNET/Security Groups restringiendo el tráfico. Auditoría con CloudTrail/Azure Monitor y SIEM (p.ej. Splunk o Azure Sentinel) para rastrear cada acceso y cambio.  
- **CI/CD**: canalizar repos a CloudFormation/Terraform con revisión obligatoria (GitOps), y escaneo automático de contenedores (Snyk/Trivy). Despliegues progresivos (canary) para minimizar impacto.  
Esta arquitectura corporativa asegura escalabilidad, monitoreo exhaustivo y cumplimiento de políticas.

## Alternativas si no existe API de SIGED  
Si SIGED no dispone de API accesible, proponemos: (a) **RPA**: scripts de automatización de UI/keyboard (Power Automate Desktop, Selenium) que extraigan documentos e índices periódicamente, simulando la consulta manual. (b) **Carga masiva**: exportación manual de expedientes (ZIP, CDX) y su ingestión con procesos batch. (c) **Integración documental**: utilizar conectores genéricos (p.ej. vía SharePoint o GEDOC si SIGED tiene WebDAV) para sincronizar archivos. En todos los casos, cada documento importado debe registrarse en la base de datos con metadatos y hash para rastreo. Debe definirse un flujo de aprobación interna para validar la entrada de expedientes cuando se realice vía RPA, evitando errores por cambios en la interfaz.

## Recomendaciones específicas para iniciar con *Bases Estándar*  
En la primera fase, recomendamos concentrarse en los TDR y las plantillas de bases. Los pasos clave son: (1) Diseñar las plantillas `.docx` (normal y abreviada) y definir claramente los campos a llenar; (2) Crear un esquema de extracción de datos del TDR (monto, alcance, duraciones, etc.), posiblemente mediante **Azure Form Recognizer** entrenado con ejemplos de TDR; (3) Implementar el flujo mínimo para “obtener TDR → extraer campos → rellenar plantilla → entregar docx”; (4) Validar internamente con usuarios funcionales para ajustar OCR/NLP y formatos de salida. Esta ruta es más acotada porque involucra una única fuente (el TDR) y pasos bien delimitados. Además, desarrolla internamente la **trazabilidad** de cada campo hacia su origen (el campo X del TDR), y establece controles de validación sencillos (por ejemplo, total base = suma de ítems). Dado que *Bases* tiene menos documentos implicados y normas más claras, permitirá demostrar valor rápidamente y sentar bases (literales) para la fase de Contratos.

## Recomendaciones para luego escalar a *Contratos*  
Una vez probado el flujo de *Bases*, el paso siguiente es abordar los Contratos, que son más complejos: involucran múltiples documentos (Bases Integradas, ofertas, garantías, etc.) y validaciones legales más estrictas. Se recomienda: (a) Reutilizar la infraestructura de OCR/NLP y de plantillas mejorada, añadiendo procesamiento para cada nuevo tipo de documento; (b) Incorporar validaciones específicas (monto contratado coincide con oferta ganadora, fechas dentro de vigencia, datos del contratista correctos) usando reglas de negocio y confirmación humana; (c) Fortalecer el RAG: indexar las Bases Integradas y cláusulas legales clave en un motor de búsqueda semántico para asistir en preguntas del tipo “¿qué garantías se exigían?”. (d) Entrenar o ajustar los modelos de extracción para nuevos campos de contratos (p.ej. datos del contrato, firmas). Es crucial mantener la revisión humana obligatoria en esta etapa por el impacto legal de los documentos. El aprendizaje del MVP (qué campos fallaban más, cómo mejorar las plantillas) se aplicará para optimizar este flujo. En esencia, escalar a Contratos implica manejar más fuentes y reglas complejas, pero la arquitectura modular inicial debe servir de base.

## Riesgos técnicos y funcionales  
- **Imprecisión de OCR/NLP**: errores en el reconocimiento (letras confusas, tablas mal leídas) pueden llenar campos incorrectos. *Mitigación*: establecer validaciones automáticas y obligar edición humana final.  
- **Alucinaciones del LLM**: el modelo generativo podría inventar datos. *Mitigación*: no confiar en generación libre, usarlo solo para extracción guiada y siempre verificar con regla o persona.  
- **Seguridad/Privacidad**: el manejo de documentos sensibles en servicios externos (OpenAI, AWS) puede violar políticas de entes públicos. *Mitigación*: usar nubes certificadas (p.ej. Azure Government) o soluciones on-premise para LLM; cifrar todo dato; separar ambientes de prueba y producción.  
- **Fallas de integración**: si SIGED cambia su esquema (URLs, credenciales), el sistema puede quebrar. *Mitigación*: abstract API con manejo de errores, logs claros y notificaciones de fallos.  
- **Sobrecarga de costos**: gran volumen de páginas OCR o tokens LLM puede disparar gasto. *Mitigación*: dimensionar con límites, usar capas de cache (extraer solo nuevos documentos), y modelos de RAG más baratos (ej. embeddings + modelo pequeño).  
- **Complejidad operativa**: demasiados componentes (microservicios, colas, índices vectoriales) aumentan puntos de falla. *Mitigación*: iniciar simple (serverless) y documentar cada módulo.  

## Preguntas clave para el equipo funcional, de infraestructura, seguridad y SIGED  
- **Funcional**: ¿Qué campos exactos se deben extraer de cada tipo de documento? ¿Cómo se validan actualmente (reglas, aprobaciones)? ¿Cuáles son las excepciones (documentos fuera de formatos estándar)? ¿Se requiere cadena de custodia o firmas digitales en los documentos generados? ¿Quiénes revisarán y aprobarán los resultados del asistente?  
- **Infraestructura/TI**: ¿Existe ambiente de nube aprobado (AWS, Azure) o restricciones a nube pública? ¿Qué topologías de red y grupos de seguridad debemos usar para aislar el sistema? ¿Qué capacidades de cómputo y GPU (para IA) se pueden provisionar? ¿Disponemos de instancias de Kubernetes o servidores dedicados para contenedores? ¿Se requiere alta disponibilidad (multi-AZ/región)?  
- **Seguridad/Compliance**: ¿Cuál es la clasificación de los documentos (confidencial, restringido)? ¿Prohíben contratos con terceros la transferencia de datos de personas naturales? ¿Existen requisitos de auditoría específicos (p.ej. retención de logs por X años)? ¿Se necesita cifrado con claves propias (BYOK)? ¿Cómo se gestiona el ciclo de vida de credenciales/API keys?  
- **Equipo SIGED/Gestión documental**: ¿Hay API REST/SOAP documentada para consultar expedientes? ¿Cuál es el volumen estimado de consultas diarias? ¿Qué mecanismos de autenticación usa SIGED? En caso de no haber API: ¿Se planifica exponer archivos (ZIP, exportaciones)? ¿Hay trabajo programado (cron) para extraer documentos? ¿Cómo se versionan y mantienen los documentos en SIGED?  

## Conclusión final  
En síntesis, la propuesta prioriza primero el asistente de *Bases Estándar* debido a su flujo acotado (un solo documento de entrada y plantillas conocidas), lo que permite probar y ajustar los componentes IA y de integración con menor riesgo. Con las lecciones aprendidas (errores de OCR, mejoras de plantilla, reglas de validación), se avanzará hacia el asistente de *Contratos*, que involucra flujos más complejos y sensibles. Se recomiendan soluciones mixtas: servicios en la nube (AWS/Azure) para acelerar el desarrollo inicial y sistemas open source locales para datos críticos. La estrategia por fases (MVP ➔ robustecimiento ➔ contratos ➔ despliegue final) reduce la complejidad y permite iterar conforme a hallazgos reales. Con esta aproximación —que incluye revisión humana obligatoria, trazabilidad completa y arquitectura segura— se busca una solución escalable, auditable y alineada con los requerimientos de entidades públicas.
