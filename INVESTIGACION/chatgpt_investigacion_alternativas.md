# Propuesta de Arquitectura de Solución IA Generativa para “Bases” y “Contratos”

## Resumen ejecutivo  
Las tendencias actuales en **Intelligent Document Processing (IDP)** muestran un crecimiento significativo (mercado proyectado USD 4.1 mil millones en 2026) con retornos de inversión típicos de 200–300% en el primer año. Para satisfacer los requisitos de la entidad pública, proponemos una arquitectura híbrida basada en nubes públicas (AWS/Azure) con posibilidad de componentes on-premise. Adoptaremos un **enfoque serverless/agil**, ya que las funciones en la nube permiten ciclos rápidos de desarrollo y despliegue con complejidad operativa reducida, mientras que los microservicios tradicionales implican costos fijos mayores. La integración con SIGED u otro sistema documentario seguirá patrones desacoplados (colas, polling, webhooks) para tolerancia a fallos. Se priorizará el cifrado y control de acceso estricto (uso de IAM con privilegios mínimos), así como trazabilidad (CloudWatch/CloudTrail) y revisión humana obligatoria para evitar errores de LLM. 

Como ruta de entrega, **fase 1** se centrará en el asistente de *Bases*, que tiene un flujo más acotado (basado en TDR y plantillas) y servirá de prueba de concepto. Luego se escalará en **fase 2** mejorando OCR/NLP y plantillas, e **fase 3** implementando el asistente de *Contratos* con más fuentes y validaciones. Finalmente, **fase 4** consolidará controles de calidad, auditoría y despliegue productivo. Este enfoque “MVP primero Bases, luego Contratos” reduce riesgos iniciales y valida componentes críticos antes de un caso de uso más complejo. En resumen, la solución propuesta combina servicios cloud (para escalabilidad y gestión) con componentes on-premise o open source según requiera la confidencialidad, optimizando costos sin sacrificar gobernanza y auditabilidad.

## Arquitectura recomendada para MVP  
Para el MVP del **Asistente de Bases**, proponemos un flujo simplificado: el usuario consulta el expediente en SIGED desde una interfaz web, el backend orquesta funciones serverless que llaman la API de SIGED, descargan el TDR, ejecutan OCR/NLP sobre dicho documento y seleccionan la plantilla de bases correspondiente. El sistema pedirá al usuario ingresar los datos administrativos (contratista, monto, plazos, etc.) mediante formularios dinámicos, y luego poblará automáticamente la plantilla .docx. Finalmente generará un documento final (docx) que se almacena en S3/Blob con un enlace de descarga pre-firmado.  

En este MVP se prioriza lo esencial: flujos serverless (por ejemplo **Lambda + Step Functions** o **Azure Functions + Durable Functions**) para agilizar iteraciones, uso de OCR automático (p.ej. Amazon Textract o Tesseract) y NLP básico (p.ej. spaCy para extraer campos técnicos). Se habilita revisión de datos extraídos antes de la generación final. El almacenamiento puede ser un bucket S3/Blob temporal con **versionado** (recuperar borradores) y plantillas en una carpeta segura. Revisiones manuales se realizarían antes de finalizar el documento.*  

## Arquitectura recomendada para producción  
Para la solución productiva se extiende el MVP con componentes robustos y escalables. Mantendremos un enfoque híbrido-cloud, privilegiando contenedores gestionados (EKS/AKS o servicios Fargate) sobre puro serverless para mayor control operacional. Se integrarán colas de mensajes (SQS o Azure Service Bus) para desacoplar subsistemas, y se usará un orquestador especializado (como **Temporal** o Step Functions) para manejar flujos complejos y reintentos. 

Además, se incorporan **bases de datos** resistentes (Aurora/Postgres o Cosmos) para estado de procesos y trazabilidad, y un **almacenamiento** enterprise (S3/Azure Blob) con políticas de ciclo de vida y retención. En la capa de IA, se habilita un motor LLM (p.ej. **Azure OpenAI GPT-4** o **Amazon Bedrock**) para tareas avanzadas de RAG y validación, apuntalado por vectores en una base (Weaviate/Milvus) para conocimiento legal y de plantillas. La imagen anterior [86] muestra el pipeline completo en un escenario AWS real; en Azure tendría componentes análogos (Logic Apps, Azure Cognitive Search, etc.). Se garantizará monitoreo con CloudWatch/Azure Monitor y auditoría de eventos clave (descarga de contrato, generación de bases), así como entornos aislados (VPC o VNET) para cumplir normativas gubernamentales.  

## Alternativas si no existe API de SIGED  
Si SIGED carece de API REST/​SOAP, se propone: (1) **Integración por extracción por lotes**: programar descargas periódicas de expedientes e importarlos a la aplicación, usando FTP o carpetas compartidas; (2) **Robotic Process Automation (RPA)**: usar herramientas como UiPath o Power Automate Desktop para automatizar la navegación web de SIGED y extraer documentos; (3) **Captura por email/scan**: configurar envío de expedientes por correo o escaneo físico a un buzón analizable. En cualquier caso, se recomienda usar patrones de **polling/colas** y registración completa de cada archivo, con manejo de errores (reintentos, quarantenas) y validación de autenticidad.  

## Recomendación específica para iniciar con Bases  
Se sugiere empezar por el flujo de *Bases Estándar* dado su alcance delimitado. Este caso solo procesa el **TDR** para extraer datos técnicos y permite escoger plantillas predefinidas, reduciendo la complejidad inicial. Al implementar primero Bases se validan los componentes clave (OCR del TDR, lógica de plantillas, UI de ingreso de datos). Esto permite ajustar modelos OCR/NLP en un contexto más simple y establecer un proceso de revisión humana antes de la generación final. Además, se puede probar localmente (p.ej. con Tesseract y python-docx) antes de migrar a servicios en la nube, minimizando costo y riesgo.  

## Recomendación específica para escalar a Contratos  
Una vez madure la solución de Bases, se integra el asistente de *Contratos*, que es más complejo: requiere agrupar múltiples documentos (Bases Integradas, ofertas, etc.), categorizarlos manualmente en el sistema, y verificar consistencias normativas/legales. Aquí conviene reforzar la extracción con RAG (usar embeddings de cláusulas estándar) y reforzar validaciones concretas de contrato (montos, plazos, entregables, datos de partes) frente a las Bases. Se debe ampliar la seguridad (p.ej. VPC dedicado) y la gobernanza (auditoría intensiva) al manejar contratos formales. La experiencia ganada con Bases facilitará este paso, pero se deberá prever más recursos de análisis (posiblemente GPUs para LLM) y pruebas con usuarios expertos en regulación antes del lanzamiento final.  

## Riesgos técnicos y funcionales  
- **Calidad de OCR/NLP:** Documentos escaneados de baja calidad o estructuras imprevistas pueden degradar la extracción de datos. Se mitigará con preprocesamiento de imagen (filtros) y ajustes de modelo.  
- **Hallucinations de LLM:** El motor generativo podría introducir información sin base; por eso requerimos sempre *prompting* con datos verificados y revisión humana final.  
- **Dependencia de terceros:** Servicios cloud pueden fallar (interrupciones AWS/Azure) o cambiar políticas. Plan de contingencia local (caché, reintentos) y arquitecturas híbridas ayudan a reducir este riesgo.  
- **Seguridad de datos:** Manejar información sensible exige cifrado extremo y segregación. Errores en IAM o encriptación podrían exponer datos confidenciales. Por ello, aplicamos políticas de menor privilegio y monitoreo ininterrumpido.  
- **Cumplimiento normativo:** Leyes de contratación pública requieren trazabilidad. Se deben generar logs detallados de todas las acciones y auditar versiones de documentos (firmas de contenido, versiones previas).  
- **Usuarios y adopción:** Un chatbot mal diseñado puede confundir; la interfaz debe ser clara y ofrecer siempre salida de texto editable. Se recomienda entrenar a los usuarios funcionales con demos y manuales.  
- **Costos ocultos:** El uso extensivo de LLM/OCR puede generar facturas altas. Es esencial monitorear tokens y llamadas, y ajustar umbrales de producción (p.ej. uso de modelos más pequeños cuando sea posible).  

## Preguntas clave a los equipos  
- **Funcional:** ¿Qué campos del TDR son **obligatorios** para las Bases? ¿Cómo se valida cada campo? ¿Quién es responsable final de cada documento? ¿Qué reglas legales deben aplicarse en cada paso? ¿Cuáles excepciones de proceso existen (p.ej. ofertas extendidas, variaciones de contratos)?  
- **Infraestructura:** ¿Disponemos de redirecciones o proxies necesarios para integrar SIGED? ¿La organización prefiere AWS o Azure (o ambas) según contratos existentes? ¿Se puede usar Kubernetes corporativo? ¿Hay requerimientos de alta disponibilidad (número de regiones)?  
- **Seguridad:** ¿Qué normativas de datos aplican (por ejemplo, datos personales, secretos estatales)? ¿Es obligatorio cifrar cada documento con llave propia? ¿Qué sistemas de SIEM/Audit logs institucional se deben integrar? ¿Quién gestionará las claves (IAM roles) y con qué separaciones de funciones?  
- **SIGED:** ¿Existe documentación o API pública de SIGED? ¿Cuál es el flujo actual de consulta de expedientes? ¿Cómo se autentica un usuario/hay que simular un rol? ¿Pueden habilitar un endpoint de prueba? ¿Cuál es el formato y metadatos disponibles de los documentos dentro de SIGED?  

## Conclusión y propuesta priorizada  
En conclusión, recomendamos un desarrollo **iterativo y seguro**: primero implementar el asistente de *Bases* con servicios cloud fáciles de prototipar (por ejemplo, AWS Lambda + Textract + React) para demostrar valor rápidamente. Luego ampliar OCR/NLP con modelos más precisos y enlazar la RAG de conocimiento interno. La fase final abarcará el asistente de *Contratos*, agregando validaciones legales y mayor control. Se enfatiza la revisión humana en cada paso para evitar hallucinaciones, y la trazabilidad completa (vía metadatos y versionado). Las decisiones tecnológicas deben priorizar escalabilidad y cumplimiento: por ejemplo, usar Azure OpenAI en producción garantiza residencia de datos y auditabilidad, mientras que implementar PostgreSQL con pgvector puede servir como vector store inicial de bajo costo. Se debe prestar especial atención a **seguridad** y **privacidad** en cada capa (encriptado, IAM) y a **observabilidad** (logs y métricas críticas). Finalmente, proponemos un **roadmap por fases** claramente definido (ver Tabla), de modo que el equipo funcional vea entregas palpables desde el primer sprint y se pueda iterar según feedback. Con este enfoque, la entidad obtendrá un asistente IA confiable y auditable que agilice sus procesos de licitación, minimizando riesgos operativos y maximizando eficiencia.
